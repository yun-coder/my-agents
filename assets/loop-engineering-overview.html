🏗️ 四层栈 —— 从 Prompt 到 Loop 的三级跳

第 4 层 · 最高
Loop Engineering（循环工程）
在 harness 之上调度 —— 怎么让它自己一遍遍跑起来
定时器跑 · 孵化小帮手 · 自我喂食 · 跨越轮次的记忆
第 3 层
Harness Engineering（工具链工程）
单次运行的武装 —— 给哪些工具、允许哪些动作、什么算完成
武装 agent 的单次运行，但不负责让它自动重来
第 2 层
Context Engineering（上下文工程）
这一刻窗口里放什么 —— 检索什么、摘要什么、清掉什么
塞错东西比塞少更糟，一窗户的噪音让再好的提示词白搭
第 1 层 · 基础
Prompt Engineering（提示词工程）
写好一次的提示词 —— 我该告诉模型什么
斟酌措辞、给例子、定角色、调语气。递进去，模型吐出来，结束。

🧩 六个零件：搭一个 Loop 需要什么

零件 01

⏱️ Automations（自动化/调度）

挂在时间表或触发器上自动跑。没有它，只是一次运行，不是 loop。

"Automations are what make a loop an actual loop."
零件 02

🌿 Worktrees（隔离）

git worktree 让每个 agent 在自己的工作目录里改代码，互不干扰。

"Two agents writing the same file = two engineers committing same lines."
零件 03

📋 Skills（SKILL.md）

固化项目知识，让 agent 每次直接取用，不必每圈重新推导上下文。

"Fire $skill-name, not a wall of instructions."
零件 04

🔌 Plugins & Connectors（MCP）

MCP 接外部系统：issue tracker、数据库、staging API、Slack。

"A loop that can only see the filesystem is a tiny loop."
零件 05

🤖 Sub-agents（生成/评判分离）

写代码的和评判的必须是两个 agent。同一个 agent 给自己的作业打分太客气。

"The model that wrote the code is way too nice grading its own homework."
零件 06

🗄️ Memory（持久化状态）

磁盘上的状态文件，跨轮跨天都在。agent 会忘，仓库不会。

"The agent forgets, the repo doesn't."

⚖️ 生成器与评判器：为什么写代码的 AI 不能给自己打分

❌ 生成器自评

"When asked to evaluate work they've produced, agents tend to respond by confidently praising the work — even when, to a human observer, the quality is obviously mediocre."

生成器的上下文中已塞满「我为什么这么写」的理由。它看自己的代码，看到的是推导过程而非结果。

✅ 独立评判器

"Tuning a standalone evaluator to be skeptical turns out to be far more tractable than making a generator critical of its own work."

借鉴 GAN 思想：generator 写，evaluator 审，结构上把「写」和「判断」彻底分开。评判器要会动手（Playwright MCP），不只是读。

🏦 Maker-Checker 原则（银行级风控）

银行里转大额，录入和复核必须是两个人。/goal 命令把这条老规矩塞进了 agent 循环：
每跑完一轮，一个又小又快的 fresh model 检查条件成立没有。不成立就再跑一轮，成立才放行。
评判器的默认态度应该是「这段代码是坏的，除非被证明能跑」。

真实案例 · 四笔账

📊 三个真实的 Loop

🌅 一、Addy 的早晨 个人级

每天早上 automation 自动醒来，读 CI 和 issue，开 worktree，子 agent 起草和审查，过了就自动开 PR， 没把握的丢进收件箱等人，状态写进文件留给第二天。一个人，一台机器，替你把脏活先过一遍。

🏢 二、Stripe 的 Minions 企业级

每周合并 1300+ PR，没有一行人手写。在 Slack 里 @ Minion bot 即可触发。 核心架构：确定性 orchestrator 备齐上下文 → LLM agent 写代码 → 硬编码 pipeline 跑 linter → agent 修 lint → git commit。 底层是开源工具 Goose 的 fork。核心论点：AI 的可靠性来自约束的质量，不是模型的大小。 沙箱用 Devbox 跑在 EC2 上，「cattle not pets」，用完即弃。

写代码 LLM agent
跑 linter / commit 硬编码 pipeline（agent 跳不过）

📉 四笔代价之账

"A loop running unattended is also a loop making mistakes unattended."
—— Addy Osmani

代价 症状 一句话防范
🔴 验证债 产出堆着没人验，错误安静积累 装一个跟干活的不是同一个的评判者
🟡 理解腐烂 代码在长，你脑里的地图停了 定期读产出，讲不出就是该更新
🟣 认知投降 循环给啥收啥，懒得有意见 执行可外包，拿主意不行
🟢 Token 失控 用量剧烈波动，账单不可预测 上线前钉死预算和重试上限

第四部分：你怎么开始

工程师身份 · 动手指南

👷 当工程师，不只是按下启动键

"Two people can build the same loop and get opposite outcomes." —— 一个人用循环是为了在吃透的事情上跑得更快， 另一个人是为了不必再去理解。循环本身不偏向任何一边，它只是个忠实的乘号。

"Build the loop. But build it like someone who intends to stay the engineer, not just the person who presses go."

循环让生成变得极其便宜，代码/方案/PR 都能批量造。稀缺的是判断力—— 知道哪个方案是对的、哪行代码该拦下来、哪个产出虽然能跑但根上错了。

✅ 第一个 Loop 检查清单

前两条决定能不能跑，后四条决定会不会闯祸。新手最容易只搭前两条就上线。

  • ✓
    发现源
    它定时去读什么？（CI / issue / commit / 收件箱）
  • ✓
    状态文件
    跨轮的记忆落在哪个磁盘文件上？
  • ✓
    Evaluator
    有没有一个独立的、会说「不」的检查？
  • ✓
    隔离
    并行的 agent 是不是各自一个 worktree？
  • ✓
    Token 上限
    设没设花费天花板？跑飞了谁拦得住？
  • ✓
    人工复核点
    哪一步停下来等你看一眼，而不是一路自动到底？
